# P2-artifacts.md — 산출물 생성 (코드 기반)

> [게이트1]에서 승인된 `qa/qa-config.yaml`을 계약으로 삼아 **케이스 인벤토리 · 런북 · 스크립트**를
> 생성한다. 종료 관문은 **[게이트2] 케이스 승인** — 기능×카테고리 커버리지 요약표를 놓고
> 사람과 공동제작 루프(다이어트/보강)를 돈다. 승인 없이 P3로 넘어가지 마라(원칙 10).

## 0. 진입 조건·산출물

- 진입: [게이트1] 통과(`qa-config.yaml`·`clearance.md` 확정). STATE.md 현재 페이즈를 P2로 갱신.
- **clearance 확보 대기와 병행한다**(원칙 12) — 담당자 회신을 기다리며 산출물 생성을 진행하라.
  단 **접근 자격이 필요한 작업(UI 크롤링 등)은 해당 clearance 항목 확보 후로 연기**한다.
- 산출물: `qa/cases/`(케이스 인벤토리) · `qa/runbook.md` · `qa/scripts/`(시딩·정리·헬스체크).
- **`qa/runner/`는 여기서 만들지 않는다.** 케이스 실행 코드화는 P4의 점진 코드화가 담당한다
  (phases/P4-retro.md) — P2에서 전면 코드화하는 선불 비용을 쓰지 마라.
- 작성 진행 커서는 발생 즉시 STATE.md에 기록(원칙 9) — 세션이 끊겨도 다음 세션이 이어 작성한다.

## 1. 케이스 인벤토리 (`qa/cases/`)

골격은 templates/case-inventory.template.md 를 채워 생성한다.

### 1.1 소스 우선순위 — 어디서 케이스를 뽑는가

| 순위 | 소스 | 비고 |
|---|---|---|
| 1 | 기존 QA/기능 문서 | 있으면 항상 최우선 — 의도된 동작의 정본 |
| 2 | API 명세 / 라우트 표면 | P0 discovery-report.md 의 추출 결과 사용 |
| 3 | UI 크롤링 | 위 둘로 부족할 때만. **접근 자격(clearance 해당 항목) 확보 후에만 수행** |

- 상위 소스가 있는 기능을 하위 소스로 대체하지 마라. 하위 소스는 상위 소스의 공백을 메울 때만.
- 각 케이스에 출처를 기록한다(1.5 포맷의 `출처` 필드) — 게이트2에서 근거를 물으면 답할 수 있게.

### 1.2 분류 A~F — 정의와 실패 우선 정렬

| 분류 | 정의 | 예 |
|---|---|---|
| **A 실패·경계** | 필수값 누락·형식 위반·경계값·중복·초과 입력에서 올바른 거부/에러 | 필수 필드 빈 값 제출 → 4xx + 에러 메시지 |
| **B 핵심 액션** | 기능의 대표 성공 경로 — 생성·수정·상태 전이가 실제로 반영되는가 | 리소스 생성 → 후속 조회로 존재 확인 |
| **C 목록·탐색** | 목록·검색·필터·페이지네이션·상세 진입 | 검색어 필터 결과가 조건과 일치 |
| **D 권한** | 역할·인증 경계 — 권한 없는 접근의 올바른 차단(**API 레벨 포함**, patterns/auth-roles.md) | 일반 역할로 관리자 API 직접 호출 → 403 |
| **E 데이터 정합** | 저장값 vs 표시값 일치. **시간 경계 체크 의무 포함** — `timezone.storage` ≠ `display`면 자정 전후·만료일 케이스 필수 | 23:50 KST 생성 건의 날짜 표시 |
| **F 교차시스템 일관성** | DB와 외부 시스템의 상태 일치 | 앱에서 생성한 리소스가 외부 시스템에도 존재 |

- **실패 우선 정렬**(원칙 3): 작성·실행 순서는 **A → B → C → D → E → F**. 정상 풀코스보다
  실패·경계·권한이 먼저다. 같은 기능 안에서도 A 케이스를 B보다 앞 번호로 둔다.
- 케이스 ID = **`<카테고리>-<번호>`**(예: `A-01`) — 번호는 카테고리 내 일련번호(01부터).
  이 ID는 `evidence/<case-id>/`·`runner/`·`known-issues.md`가 상호참조하는 **공통 키** —
  확정 후 변경 금지, 케이스를 제외해도 번호를 재사용하지 마라.

### 1.3 커버리지 규칙

- **최소선**: `scope.priority_features`의 **우선 기능마다 A·B·D 각 최소 1케이스**
  (A에는 필수값 누락·경계값 중 해당되는 것을 우선 배치 — principles.md 원칙 3).
- **의무 조건부**: `timezone.storage` ≠ `display`면 E의 시간 경계 케이스 의무. 외부 시스템이
  있으면(F 대상) 핵심 연동마다 F 최소 1케이스.
- **총량 상한**: 전체 케이스 수 ≤ `scope.case_cap`. 초과하면 C(목록·탐색) → 중복 성격의
  B/E 순으로 감축하되 **A·B·D 최소선은 절대 깎지 마라**. 상한 자체를 바꾸고 싶으면
  임의 변경 금지 — 게이트2에서 사람에게 상정한다(원칙 10).
- `scope.excluded`에 든 기능·라우트는 케이스를 만들지 않는다(제외 사유는 인벤토리에 주석).

### 1.4 기대값(오라클) 출처 우선순위

| 순위 | 출처 | 규칙 |
|---|---|---|
| 1 | 기능 문서 | 명세된 기대값이 정답 |
| 2 | API 스키마 | 문서가 없으면 스키마의 타입·필수 여부·상태코드가 기준 |
| 3 | 현행동작 스냅샷 | 둘 다 없을 때 최후 수단 — 케이스에 **"회귀 기준(정답 아님)"** 을 반드시 명시 |

- 스냅샷 기반 오라클은 "지금 이렇게 동작한다"의 기록일 뿐이다 — **현재 버그가 정답이 되는 것을
  방지**하기 위해 표기를 강제한다. 스냅샷 케이스의 FAIL은 "회귀 발생 또는 원래 버그" 두 해석이
  가능함을 리포트에서 유지한다.
- 각 케이스에 `오라클 출처` 필드를 기록한다(1.5 포맷).

**드리프트 규칙 — 상위 소스가 코드보다 낡았을 때**: 기능 문서/명세(1순위)가 현재 코드·스키마·
마이그레이션과 **모순**되면(예: 문서가 재설계 이전 상태), 그 기능에 한해 **코드/스키마를 오라클로
승격**하고 케이스 `오라클 출처`에 `문서 stale — 코드 기준`이라고 표기한다. 문서 전체를 버리지는
말고(다른 기능은 여전히 유효), 모순난 부분만 강등한다. 모순 사실은 STATE 결정 로그에 남긴다
— "문서가 낡았다"는 그 자체로 리포트할 가치가 있는 발견이다.

**바이너리 스펙 소스**(.xlsx·.pdf 등 열 수 없는 기능 문서): 정적 분석으로 내용을 못 읽으면
1순위를 **자동 강등**하지 말고 — ① clearance 항목으로 등록해 "텍스트/CSV 내보내기 또는 열람 도구"를
요청하고 ② 확보 전까지는 2순위(API 스키마)로 잠정 진행하되 케이스에 `오라클 잠정 — 스펙 확보 후 재확인`
표기 ③ 확보되면 해당 케이스 오라클을 재검토한다. 열 수 있는 도구(예: 표 변환)가 능력 프로파일에 있으면 먼저 시도한다.

### 1.5 케이스 포맷

케이스 본문은 4단계 고정: **`[세팅] → [실행] → [실제값 검증] → [정리]`**

| 필드 | 내용 |
|---|---|
| ID / 제목 | `<카테고리>-<번호>` + 한 줄 제목 |
| 출처 | 1.1의 소스(문서명·API 경로·크롤링) |
| 오라클 출처 | 1.4의 출처 — 스냅샷이면 "회귀 기준(정답 아님)" 병기 |
| 환경 제약 | 실행 가능 환경(`environments`의 키 이름). prod 등급 환경은 읽기전용 케이스만 |
| [세팅] | 전제 상태 만들기 — **API 시딩 우선**(patterns/seeding.md), 브라우저는 검증할 UI에만 |
| [실행] | 도구 중립 스텝 어휘로 기술(§2와 동일 어휘) |
| [실제값 검증] | 기대값과 **구체적 실제값** 대조 — 네트워크(상태코드·응답 body)·DB·외부 시스템 교차(원칙 2). "에러 안 남"은 검증이 아니다 |
| [정리] | 생성 리소스 삭제 + **잔여 0건 검증**(원칙 5). 생성이 없으면 "정리 없음" 명기 |
| 증거 요구사항 | PASS 판정에 필요한 증거 파일 목록 — `evidence/<case-id>/`에 남길 것(스키마: patterns/evidence.md) |

- 시딩 데이터는 `data.test_prefix` + run-id 명명(원칙 4), `data_boundary` 안에서만 생성,
  `risk.irreversible_actions`·`data.deny_list` 대상은 케이스 스텝에 넣지 마라.

## 2. 런북 (`qa/runbook.md`)

골격은 templates/runbook.template.md 를 채워 생성한다.

- 스텝은 **도구 중립 어휘**로만 기술한다: `navigate` / `fill` / `click` / `expect-network` /
  `assert-db` … — 정의와 **두 바인딩(대화형 브라우저 도구 ↔ Playwright) 대응표**는
  patterns/browser.md. 런북이 도구 중립이어야 어떤 에이전트든 같은 런북을 실행할 수 있다.
- 특정 도구의 함수명·커맨드를 런북에 직접 쓰지 마라. 어휘에 없는 동작이 필요하면 가장 가까운
  어휘 + 주석으로 기술하고, 어휘 추가 제안은 `qa/kit-feedback.md`에 남겨라(킷 수정 금지).
- 케이스는 1.2의 정렬 순서로 배치하고, 수동 개입이 필요한 지점(유인 로그인·2FA 등,
  patterns/auth-roles.md)을 스텝에 명시한다 — P3 사전 고지의 근거가 된다.

## 3. 스크립트 (`qa/scripts/` — Node.js 표준)

- 런타임은 **Node.js 고정**. bash 보편성을 가정하지 마라(Windows 호스트 대비).
  시크릿은 `.env.qa`에서 `secret_ref` 키 이름으로만 읽는다 — 코드·로그에 실값 금지(원칙 6).
- **시딩 스크립트** (상세: patterns/seeding.md)
  - 완료 조건은 **생성 API 200이 아니라 후속 조회 성공**이다 — 비동기 반영·지연을 가정하고
    **readiness 폴링 + 타임아웃 + 백오프를 내장**하라. 폴링 실패는 시딩 실패로 처리한다.
  - 생성한 리소스는 즉시 STATE.md **시딩 잔여물 대장**에 기록 — 세션이 죽어도 다음 세션이 치운다.
  - 생성 건수는 `risk.budget_per_run.resource_creations` 상한 안에서.
- **정리 스크립트** — **정리검증 내장**: 삭제 호출로 끝내지 말고, run 중 생성 인벤토리
  (잔여물 대장)와 대조해 **DB 잔여 0건 + 외부 시스템 잔여 0건**을 확인 쿼리·조회로 검증하라.
  삭제 방식은 프로젝트 정책을 따른다(soft-delete가 정책이면 soft — 원칙 5).
- **헬스체크 스크립트** — 대상 환경 `base_url` 응답 확인. 모든 run의 프리플라이트에서 쓰인다.
- 장시간 루프(폴링 등)에는 **킬스위치 체크를 내장**하라 — 루프마다 `qa/.stop` 존재를 폴링,
  발견 시 즉시 안전 중단. 실행 상호배제는 `qa/.lock`(stale 판정 30분) 규약을 따른다.

## 4. [게이트2] 케이스 승인 — 공동제작 루프

templates/gate-summary.template.md 고정 형식으로 제시하고 **멈춘다**. 제시물 2가지:

1. **기능×카테고리 커버리지 요약표** — 행=`priority_features`, 열=A~F, 셀=케이스 수.
   비전공자가 **빈 칸(부족)과 과밀(과잉)을 눈으로 판단**하는 표다. A·B·D 열의 0은 규칙 위반으로
   표시하고, 총량(합계 / `case_cap`)을 함께 보여라.

   | 기능 \ 분류 | A | B | C | D | E | F |
   |---|---|---|---|---|---|---|
   | `<기능1>` | 2 | 1 | 1 | 1 | 1 | — |

2. **케이스 목록** — ID · 제목 · 출처 · 오라클 출처(스냅샷이면 "회귀 기준" 표기) · 환경 제약.

**공동제작 루프**: 응답이 "수정: <내용>"이면 —
- **다이어트**(과하다): 지목 영역을 감축하되 A·B·D 최소선 유지. 최소선과 충돌하면 그 사실을
  알리고 재확인을 받아라.
- **보강**(부족하다): 빈 칸을 채우되 `case_cap` 초과 시 상한 조정 여부를 함께 상정.

수정 반영 → 요약표 재제시 → **"승인"까지 반복**. 루프에서 합의된 범위·상한 변경은
`qa-config.yaml`에 반영하고 결정 로그에 남긴다(원칙 10 — 임의 변경 금지, 이 게이트가 승인점이다).
