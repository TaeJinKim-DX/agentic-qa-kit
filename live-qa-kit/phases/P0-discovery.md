# P0 — Discovery (자동 · 질문 금지)

> 목적: 사람에게 묻기 전에 **코드·설정을 근거로** 프로젝트를 전수 파악하고, 자신의 능력을 탐지하고,
> 위험·준비물 후보를 초안으로 만든다. 산출물은 `qa/discovery-report.md` — P1 인터뷰의 재료다.
> 발견 가능한 것을 P1에서 묻는 것은 실패다(P1 질문 원칙: 발견 가능한 건 묻지 않는다).

## 0. 규칙 (P0 전체에 적용)

- **질문 금지. 예외: 없음.** 모르는 것·확정 못 한 것은 전부 discovery-report의 **[파악 못 한 것]**에
  {항목, 시도한 것, 알아내려면 무엇이 필요한가}로 기록하라. 그 목록이 P1 질문의 재료가 된다.
  (모노레포 등 프로젝트 루트 후보가 복수인 경우도 동일 — 기록하고 P1에서 확인. AGENT.md §1)
- **관찰만 한다.** 프로젝트 코드 수정 금지. 대상 시스템에 **non-GET 요청 금지**(부작용 방지).
  유일한 예외는 §5 기동 시도(로컬 프로세스 기동·종료)다.
- **탐지는 선언이 아니라 실행으로 검증한다.** "될 것 같다"가 아니라 실제 커맨드 1회 성공을 근거로 기록.
- 시크릿을 발견해도(설정 파일의 키·토큰 등) **값을 어디에도 옮겨 적지 마라**(원칙 6).
  위치와 키 이름만 기록한다(이후 qa-config의 `secret_ref` 후보).

**시작 시 워크디렉터리 초기화** (없으면):

1. 프로젝트 루트(킷 폴더의 부모) 직하에 `qa/` 생성.
2. `qa/STATE.md`를 `templates/STATE.template.md`로 생성 — `kit_version`(킷 CHANGELOG.md 최신),
   현재 페이즈 = P0 기입.
3. `qa/.gitignore` 생성: `.env.qa` `.auth/` `.lock` `.stop` 제외 등록.

**델타 모드** (P5 순환에서 재진입한 경우만): 전체를 다시 하지 않는다. §1의 **라우트/API 표면**만 재수집해
STATE.md의 "마지막 디스커버리 기준"(커밋 SHA/앱 버전) 시점 스냅샷과 diff → 신규·변경·삭제 라우트를
커버리지 갭 리포트로 산출하고 phases/P5-unattended.md의 절차로 복귀한다. 이하 §2~§5는 생략.

## 1. 파악 항목 체크리스트 (코드·설정 근거)

8개 항목을 순서대로. 각 항목은 **근거 파일 경로**와 함께 기록한다. 확인 불가 항목은 [파악 못 한 것]으로.

| # | 항목 | 확인 방법(단서) | 기록할 것 |
|---|---|---|---|
| 1 | 스택 | 매니페스트(package.json, build.gradle, pom.xml, requirements.txt, go.mod 등)·lock 파일·프레임워크 설정 | 언어·프레임워크·버전·패키지 매니저, 프론트/백 분리 여부 |
| 2 | 기동법 | README·매니페스트 scripts·Dockerfile/compose·Makefile·CI 설정 | 기동 커맨드, 포트, 선행 조건(DB 기동·env 파일·마이그레이션) |
| 3 | 라우트/API 표면 | 라우터 정의·컨트롤러 어노테이션·OpenAPI/Swagger 스펙·프론트 라우터 | 전체 엔드포인트 표(메서드·경로·요약) — §3 deny-list 추출의 입력 |
| 4 | 인증 | 인증 미들웨어·로그인 폼/컨트롤러·세션/JWT 설정·역할 정의 | 방식(form/sso/token), 2FA·캡차 유무, 역할 목록, 세션 수명 단서 |
| 5 | DB | 접속 설정·ORM/마이그레이션 파일·스키마 정의 | 종류·스키마 개요·QA가 조회할 수단(클라이언트·포트) 존재 여부 |
| 6 | 외부 시스템 | HTTP 클라이언트 코드·SDK 의존성·webhook·메일/스토리지/외부 API 설정 | 시스템 목록 + 각각의 부작용(발송·생성·과금) — §3·§4의 입력 |
| 7 | 기존 QA/기능 문서 | test/ 디렉터리·QA 문서·기능정의서·API 명세·설계 문서 | 경로 목록 — P2 케이스 인벤토리의 1순위 소스 |
| 8 | 저장 vs 표시 타임존 | DB 컬럼 타입(timestamptz 등)·서버 TZ 설정·프론트 날짜 포맷 코드 | 저장 TZ / 표시 TZ — 다르면 시간 경계 케이스(E) 의무 대상 |

- 표면(#3)은 **전수**를 지향하되, 자동 생성 스펙(OpenAPI 등)이 있으면 그것을 정본으로 삼고 코드와 대조하라.
- #4~#6에서 발견한 프로젝트 고유 토큰·키 형식은 **secret_patterns 후보**로 별도 기록(§3).
- 환경 판별 신호(헬스 응답의 env 식별값 등) → qa-config `environments.<env>.fingerprint` 후보로 기록(P1에서 확정).

## 2. 능력 탐지 (capability profile)

5개 능력을 탐지해 **STATE.md의 "능력 프로파일 (P0 탐지)" 표에 즉시 기록**한다.
값 어휘는 STATE.template.md의 해당 표를 그대로 따른다.

### 2.1 브라우저 자동화

1. **대화형 브라우저 도구**가 자신의 도구 목록에 있는가 → 있으면 값 = "대화형 브라우저 도구". 끝.
2. 없으면 **Playwright 경로**: Node.js 확인(§2.5 선행) → `playwright` 설치 →
   브라우저 바이너리 설치 시도. 다운로드 불가(프록시·차단)면 시스템 크롬 폴백(`channel: 'chrome'`).
   상세 커맨드·폴백 순서는 patterns/browser.md.
3. **검증: 빈 페이지(`about:blank`) 스크린샷 1장 성공**이 확보 기준이다. 설치 로그 성공만으로 확보로 치지 마라.
4. 둘 다 실패 → 값 = "없음(강등)". 브라우저 케이스는 수동 표기 예정·API 케이스는 계속 —
   강등 규칙은 patterns/browser.md. 실패 원인은 [파악 못 한 것] + 필요 시 준비물 후보(§4)로.

### 2.2 구조화 질문 UI

자신에게 구조화 질문 도구(선택지 제시 UI)가 있는지 확인. 없으면 "없음(채팅 번호 질문)" — P1에서
번호 매긴 채팅 배치를 쓴다.

### 2.3 헤드리스 재호출 수단

자신을 비대화형(스크립트에서 1회 호출)으로 재호출할 커맨드가 있는가. 있으면 **호출 커맨드 전문**을 기록
(P5의 "에이전트 회고" 선택 계층에 사용). 실제로 무해한 1회 호출로 검증할 수 있으면 검증하라.

### 2.4 스케줄 수단

사용 가능한 것 전부 나열: cron / OS 작업 스케줄러 / 에이전트 자체 스케줄 기능.
존재 확인 커맨드 1회로 검증(등록은 하지 마라 — 스케줄 등록은 게이트4 이후에만, 원칙 7).
셋 다 없으면 "없음(반무인 강등)" — P5에서 사람이 트리거하는 반무인으로 강등된다.

### 2.5 Node.js

`node -v` 실행으로 버전 확인. Node.js는 스크립트 표준 런타임(P2)이다.
없으면 설치 가능 여부까지 확인하고, 불가면 [파악 못 한 것] + 준비물 후보(설치 권한/대안)로 기록.

## 3. deny-list 후보 자동 추출

§1-#3의 라우트 표면에서 **부작용 있는 non-GET(POST/PUT/PATCH/DELETE)을 전수 나열**하고 분류한다.

1. non-GET 전체 표: 메서드·경로·하는 일·부작용 대상(DB/외부 시스템/알림).
2. **불가역 후보** 표시: 삭제·발송(메일/알림)·결제·외부 리소스 생성·권한 변경 등 되돌릴 수 없거나
   외부로 새는 것. → deny-list 후보 초안(라우트 패턴 형식 — qa-config `risk.irreversible_actions`에 들어갈 모양으로).
3. **UI 휴리스틱 후보**: 화면 코드에서 발견한 파괴적 액션 버튼 텍스트(삭제·발송·확정류) —
   대화형 경로의 클릭 전 검사에 쓰인다.
4. §1에서 모아둔 **secret_patterns 후보**(프로젝트 고유 토큰 형식)도 여기에 병기.

이 초안은 **P1에서 사용자가 확정**한다(P0에서 확정하지 마라 — 원칙 10). discovery-report [위험 후보]에 수록.

## 4. 준비물 후보 전수 예측 (원칙 12)

코드·설정을 근거로 **P2~P5 전 과정**에 필요한 자격·접근·허가·정보를 전수 예측한다.
목적: 테스트 도중 "권한이 없어서 대기"로 멈추는 일 방지 — 빠뜨린 준비물 1개가 P3를 통째로 세운다.

5분류 전부에 대해 후보 유무를 판단하라(해당 없음도 "해당 없음"으로 기록):

| 분류 | 예측 단서 → 후보 |
|---|---|
| 계정 | 인증·역할 정의(§1-#4) → 역할별 테스트 계정(권한 케이스 D의 축) |
| 키 | 외부 API 클라이언트·SDK 설정(§1-#6) → API 키·시크릿(이후 `secret_ref`로 등록) |
| 접근 | 내부망 URL·프록시 설정 → VPN·방화벽·프록시 개통 |
| 외부권한 | 외부 시스템에 쓰기 동작(예: 저장소·리소스 생성) → 해당 시스템의 생성·정리 권한 |
| 승인 | 스케줄 운용(P5)·데이터 정리 정책·prod 인접 환경 사용 → 담당자 승인 항목 |

후보마다 {무엇이, 왜(어느 페이즈·어느 케이스 카테고리에 필요), 요청 대상 추정}을 기록 →
**clearance 초안**으로 discovery-report에 수록. P1에서 `qa/clearance.md`(정본 대장, CL-## ID·복붙 요청 문구)로
확정한다(templates/clearance.template.md).

## 5. 기동 시도 (타임박스 10분)

§1-#2의 기동법을 실증한다. **타임박스 기본 10분 — 시작 시각을 기록하고 지켜라.**

1. 파악한 커맨드로 로컬 기동 시도 → 헬스 URL(또는 루트)에 GET 1회로 확인.
   브라우저 능력이 있으면 첫 화면 1장 캡처(선택).
2. **성공**: 기동 커맨드·포트·헬스 URL·선행 조건을 기록하고, 띄운 프로세스를 종료한다.
3. **실패 또는 타임박스 초과**: 그 자리에서 멈춘다. **디버깅에 빠지지 마라.**
   원인·에러 요지·기동에 필요해 보이는 조건(DB·env·의존 서비스)을 [파악 못 한 것]에 기록하고 **즉시 P1로**.
   P1에서 대상이 배포 환경으로 정해지면 로컬 기동은 애초에 불필요 — URL 헬스체크로 대체된다.

## 7. 산출물 — `qa/discovery-report.md` 목차

이 목차 순서·제목 그대로 작성한다.

1. **요약** — 3줄: 무엇을 파악했나 / 못 한 것 몇 건 / 다음 단계(P1)
2. **파악한 것** — §1의 8항목 표(근거 파일 경로 병기)
3. **파악 못 한 것** — 항목 / 시도한 것 / 알아내려면 필요한 것(→ P1 질문 후보)
4. **위험 후보** — deny-list 초안(non-GET 표·불가역 표시), UI 휴리스틱 후보, 외부 부작용 목록,
   prod 오인 위험 단서, secret_patterns 후보
5. **능력 프로파일** — §2 결과(정본은 STATE.md — 여기는 사본 명시)
6. **준비물 후보 (clearance 초안)** — §4의 5분류 표
7. **기동 시도 기록** — 커맨드·결과·소요 시간
8. **P1 인터뷰 후보 답** — 필수입력형 질문에 미리 제시할 후보(예: base_url 후보, 역할 목록,
   deny-list 후보) — P1의 "후보 답 제시" 재료

작성 후: phases/P1-interview.md 를 읽고 P1을 시작하라.
