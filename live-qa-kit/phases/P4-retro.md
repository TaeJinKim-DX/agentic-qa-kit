# P4 — Retro & Hardening (회고·자가진단·점진 코드화)

> 직전 run의 실패를 전수 분류해 개선하고, 안정화된 케이스를 `qa/runner/`로 코드화한다.
> P4 자체에는 게이트가 없다 — 안정화 기준을 만족할 때까지 **P3 ↔ P4를 반복**하고,
> 만족하면 P5(phases/P5-unattended.md) [게이트4] 상정 자격이 생긴다.
> 입력: `runs/<env>-<run-id>/report.md` + `evidence/<case-id>/` + STATE.md.

## 1. 실패 전수 분류

report.md의 모든 FAIL을 하나도 남기지 말고 분류하라. 분류 불가면 **"미분류"로 명시 기록**한다
(숨기면 클린 런 판정이 오염된다 — 미분류는 클린 런을 차단한다, §5).

**선별 먼저 — AUTH 이슈**: 로그인 실패·세션 만료로 인한 실패는 **케이스 실패로 세지 않는다.**
AUTH 이슈로 별도 분류하고, patterns/auth-roles.md 절차로 세션을 재확보한 뒤 해당 케이스를
재실행 대상으로 표기하라. 케이스 자체의 FAIL 통계에 섞이면 3분류 전체가 왜곡된다.

나머지 실패는 3분류:

| 분류 | 판정 | 처리 |
|---|---|---|
| **제품 버그** | 대상 시스템이 기대와 다르게 동작 — §2 오탐 방지 3관문 통과 필수 | `qa/known-issues.md` 등록(등록은 [게이트3] 승인 사항 — §3) + 케이스에 **expected-fail** 표기 + 리포트 이관 |
| **테스트 버그** | 케이스·스크립트·셀렉터·대기 전략의 결함 | **즉시 수정**(기술 세부 — 에이전트 직접, 원칙 10) |
| **문서 갭** | 런북·config가 현실과 달라 실행이 어긋남 | `qa/runbook.md` 직접 개선. qa-config.yaml 변경은 정책 변경 — §7 |

**플래키**: 같은 케이스가 run마다 PASS/FAIL이 흔들리면 플래키로 집계하고(run-report 정량 필드),
원인(대기·타이밍·데이터 오염)을 찾아 테스트 버그로 처리하라. 플래키는 §4 편입 기준의
"연속 2회 PASS"를 끊는다 — 방치하면 코드화가 영원히 안 된다.

## 2. 오탐 방지 3관문 (제품 버그 주장 전 필수 — 원칙 8)

세 관문을 **순서대로** 다 통과해야 "제품 버그"라 부를 수 있다:

1. **필드 실존**: 이상한 값을 읽은 그 필드/엔드포인트가 스키마·코드에 실제로 존재하는가?
   (없는 필드를 읽고 "값이 비었다"고 오판한 실사고가 두 번 있었다)
2. **의도된 설계**: 설계 문서·기능 문서·결정 기록을 확인하라 — 의도된 동작일 수 있다.
3. **재현 2회**: 같은 조건에서 2회 재현하라. 재현 증거는 `evidence/<case-id>/`에 남긴다.

관문에서 걸리면 제품 버그가 아니다 — **오탐으로 계상**하고(retro 정량 필드) 테스트 버그
또는 문서 갭으로 재분류하라. 리포트에는 판정 근거(어느 관문을 어떻게 통과/기각했는지)를 함께 적는다.

## 3. known-issue 등록 (조용히 묻지 않기)

- 3관문을 통과한 제품 버그는 `qa/known-issues.md`(templates/known-issues.template.md) 등록 대상.
  **등록은 [게이트3] 승인 사항이다** — 이번 run의 게이트3에서 승인된 건만 등록하고,
  P4 중 새로 판정된 건은 "등록 제안"으로 다음 게이트3에 올려라. 승인 없이 등록 금지.
- 등록된 건은 해당 케이스에 **expected-fail**을 표기한다 — 이후 run에서 그 실패는
  "예상된 실패"로 리포트되고, 클린 런을 차단하지 않는다(§5).
- expected-fail 케이스가 갑자기 PASS하면 그것도 이벤트다 — "수정된 듯, known-issue 해소 확인 요망"으로
  리포트하고 해소 처리도 게이트에 올려라.

## 4. 점진 코드화 — runner/ 편입

P2에서 전면 코드화하지 않은 이유가 여기 있다: **안정화가 증명된 케이스만** 코드로 굳힌다.

- **케이스별 편입 기준: 유인 연속 2회 PASS + 그 사이 셀렉터 무수정.**
  둘 중 하나라도 미달이면 그 케이스는 유인 런에 남는다. 기준을 낮춰 편입하지 마라 —
  무인 신뢰성이 여기서 결정된다.
- **무인 범위 = 러너 편입 케이스만**(P5). 미편입 케이스는 계속 유인 런으로 돈다.
- 러너 요건(`qa/runner/`, Node.js — patterns/browser.md의 Playwright 바인딩 사용):
  - run-all 실행·판정·리포트·정리검증까지 에이전트 없이 완결.
  - 케이스 ID(`<카테고리>-<번호>`, A-01 식)를 그대로 유지 — cases/·evidence/·known-issues의 공통 키.
  - 증거는 유인 런과 동일 스키마로 `runs/<env>-<run-id>/evidence/<case-id>/`에(patterns/evidence.md, redaction 포함).
  - 킬스위치 `qa/.stop` 체크(케이스 경계마다 + 내부 루프 폴링) · 실행 락 `qa/.lock`(stale 30분) 준수.
- 편입/제외 이력은 STATE.md 결정 로그에 즉시 기록하라(원칙 9).

## 5. 클린 런 · 안정화 판정

- **클린 런 = 테스트 버그 0 + 문서 갭 0 + 미분류 실패 0.**
  known-issue로 등록된 제품 버그(expected-fail)는 클린 런을 **차단하지 않는다** —
  제품 버그가 고쳐질 때까지 QA가 안정화 못 하는 데드락을 막기 위한 정의다.
- **안정화 기준 = 러너 기준 연속 2회 클린 런 → 무인 준비 완료.**
  "러너 기준"이다 — 유인 런의 클린은 세지 않는다. 무인에서 돌 코드가 스스로 2회 증명해야 한다.
- 미달이면: 이번 P4의 수정·편입을 반영해 **P3를 재실행**하라(phases/P3-first-run.md).
  이 반복이 이 킷의 자가개선 루프다.

## 6. retro.md 작성

templates/retro.template.md를 채워 `runs/<env>-<run-id>/retro.md`로 생성하라. 필수 내용:

- 실패 3분류 집계 + AUTH 이슈 건수(별도) + 미분류 건수.
- **오탐 수·오탐률 추세**(3관문에서 기각된 버그 주장 수 — 이전 run들과 비교),
  **플래키율 추세**. 추세가 나빠지면 원인 분석을 함께.
- 러너 편입 현황(편입 n / 전체 m, 이번에 편입/보류된 케이스와 사유).
- 클린 런 여부 + 안정화 카운터(연속 클린 런 몇 회째).
- **킷 개선 제안**: 킷 자체의 결함·개선점은 retro의 해당 섹션에 적고 `qa/kit-feedback.md`로 옮겨라.
  킷(live-qa-kit/ 하위)은 수정 금지 — 사람이 킷 저장소에 반영한다(AGENT.md §0).

## 7. 정책 변경은 자동 적용 금지 (원칙 10)

P4에서 에이전트가 직접 하는 것과 게이트에 올리는 것을 구분하라:

| 직접 처리(기술 세부) | 다음 게이트 상정(정책 변경) |
|---|---|
| 셀렉터 수정·대기시간 보정·스크립트 버그 수정 | 케이스 **추가/제외** |
| 런북 절차 교정(문서 갭) | 판정 **기준 변경**(오라클·커버리지 규칙) |
| 러너 편입(§4 기준 충족 시 — 이력은 STATE 기록) | known-issue **등록/해소**([게이트3]) |
| retro.md·kit-feedback.md 작성 | **qa-config.yaml 변경**(계약 — 템플릿 헤더 규칙) |

애매하면 정책 변경으로 취급하고 게이트에 올려라.
