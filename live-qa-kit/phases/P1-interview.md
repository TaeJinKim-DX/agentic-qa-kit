# P1 — Interview (맞춤 설정)

> **목적**: P0 발견을 바탕으로 꼭 필요한 것만 묻고, 답을 계약(`qa/qa-config.yaml`)과
> 준비물 대장(`qa/clearance.md`)으로 만든다. 끝은 **[게이트1] 설정 승인**.
> 입력: `qa/discovery-report.md` · 산출: `qa-config.yaml` + `clearance.md` + STATE.md 갱신.

## 0. 진입 조건

- `qa/discovery-report.md`가 있어야 한다. 없으면 P0부터(phases/P0-discovery.md).
- STATE.md의 현재 페이즈를 P1로 갱신하고 시작한다.
- P0에 "루트 후보 복수"가 기록되어 있으면 ENV-00이 첫 질문이다(AGENT.md §1의 예외적 허용 질문).
- **diff 인터뷰 모드**: `kit_version`/`config_schema_version` 불일치로 진입했다면 —
  바뀐 질문만 묻고 config를 증분 갱신한다. 전체 재인터뷰 금지(AGENT.md §2).

### 0.1 사용자 부재 모드 (무인 프리빌드 · 도그푸딩 · CI)

응답할 사람이 지금 없이 산출물만 미리 만들어 둬야 하는 상황(프리빌드)에서는:

- **질문을 사람에게 던지지 않는다.** 대신 질문 은행을 다음 규칙으로 자체 채운다:
  - `[기본값형]` → 추천 기본값을 적용하고 qa-config 필드에 `# 프리빌드 — 기본값 적용` 주석.
  - `[필수입력형]` 중 사실·자격·접근 항목 → **전부 clearance.md 미확보로 라우팅**(지어내지 마라).
    보수값이 성립하는 정책 질문만 보수 설정 + 주석(§2 3보조와 동일).
- **게이트는 건너뛰지 않고 "상정-정지"한다**(원칙 10 유지): 게이트 요약(gate-summary 형식)을
  생성해 `qa/`에 남기고, STATE.md에 `게이트N: 상정 — 사람 승인 대기`로 기록한 뒤 **멈춘다**.
  승인을 자작하지 마라. 사람이 오면 그 요약으로 승인을 받는다.
- 진입 신호: 사용자가 "사람 없이 준비만", "프리빌드", "도그푸딩"이라고 지시했거나, 대화 채널에
  응답자가 없는 무인 실행이다. 애매하면 기본은 유인 모드(사람에게 묻는다).
- **P3(라이브 실행)으로는 절대 자동 진입하지 않는다** — 프리빌드의 종착은 P2 산출물 + 게이트 상정까지다.

## 1. 질문 원칙 4

1. **발견 가능한 건 묻지 않는다.** discovery-report에 답이 있으면 질문하지 말고
   "확인 항목"으로 바꿔라 — 답을 미리 채워 제시하고 맞는지만 확인받는다.
2. **판단 질문엔 반드시 추천 기본값.** 기본값 없는 판단 질문은 내보내지 마라.
   사용자가 "다 추천대로" 한 마디로 기본값형 전부를 수용할 수 있어야 한다.
3. **카테고리별로 한 번에 배치한다.** 낱개로 찔끔찔끔 묻지 마라.
   필수입력형은 최소화해 배치 **맨 앞에 별도 표기**로 모은다.
   질문 UI는 능력 프로파일을 따른다 — 구조화 질문 도구가 있으면 사용, 없으면 채팅에 번호 매긴 배치(AGENT.md §5).
4. **모든 답은 qa-config.yaml에 기록한다.** 채팅에만 남은 답은 없는 것과 같다(원칙 9).
   질문마다 기록 위치(필드)가 정해져 있다 — 아래 질문 은행의 `기록:` 참조.

**수량 목표**: 실제 내보내는 질문 총 ≤ 15개, 그중 직접 입력 요구(필수입력형) ≤ 3개.
은행 전체를 그대로 다 묻는 것은 원칙 1 위반이다 — P0 발견으로 최대한 "확인 항목"으로 줄여라.

## 2. 질문 태그 체계

| 태그 | 뜻 | 사용자가 할 일 |
|---|---|---|
| `[기본값형]` | 판단 질문 — 전부 추천 기본값이 있다 | 기본값 수용("다 추천대로" 가능) 또는 다른 값 지정 |
| `[필수입력형]` | 사실 질문 — 대상 URL·테스트 계정 등 | 직접 답한다(아래 3보조로 부담 최소화) |

### 필수입력형 3보조 (모든 필수입력형에 적용)

1. **후보 답 제시**: P0 발견을 근거로 후보 답을 먼저 제시한다
   (예: "코드에서 메일 발송 3곳 발견 — deny-list 후보"). 사용자는 고르거나 고치기만 하면 된다.
2. **"모르겠다" 라우팅** — 두 갈래를 혼동하지 마라:
   - **보수적 값이 존재하는 판단 성격 질문**(데이터 경계·정책류)만 → 가장 보수적 설정을
     적용하고, qa-config 해당 필드에 `# "모르겠다" — 보수 설정 적용` 주석을 남긴다.
   - **자격·접근·정보성 항목**(계정·키·환경 접근·담당자만 아는 사실) → 보수값이 성립하지 않는다.
     **clearance.md 미확보 항목으로 라우팅**한다 — [게이트1]은 통과 가능하지만 P3 진입은 차단된다
     (phases/P3-first-run.md).
3. **담당자 전달 문구 생성**: 사용자가 답을 모르면 "담당자에게 이렇게 물어보세요" 복붙 문구를
   만들어 clearance.md 부록에 넣는다. 문구에는 {무엇이 필요한가 / 왜(어떤 QA 목적) /
   언제까지 / 회신 형식}을 담고, 시크릿 값은 채팅·메일이 아니라 팀의 안전한 전달 수단으로
   받도록 명시한다(templates/clearance.template.md 부록 규칙).

## 3. 질문 은행

사용 규칙:
- `<…>`는 discovery-report의 발견값으로 채워 내보낸다. 발견된 항목은 질문이 아니라 확인 항목으로.
- 질문 ID는 은행 관리용이다(케이스 ID `<카테고리>-<번호>`와 별개 체계). 배치에 내보낼 때는
  번호만 다시 매겨도 된다.
- 시크릿 실값은 어떤 질문에서도 받지 마라(원칙 6). config에는 `secret_ref`(.env.qa 키 이름)만,
  실값은 사용자가 gitignored `qa/.env.qa`에 직접 기입한다.

### 3.1 환경

**ENV-00 `[필수입력형]` (조건부 — P0에 루트 후보 복수 기록 시에만)**
- 질문: "프로젝트 루트 후보가 여러 개입니다: <후보 목록>. `qa/` 워크디렉터리를 어디에 만들까요?"
- 왜: 위치가 어긋나면 STATE 탐색·재개가 깨진다(AGENT.md §1).
- 보조: P0 근거로 권장 후보를 지정해 제시.
- 기록: STATE.md 결정 로그(위치 확정).

**ENV-01 `[필수입력형]` — 대상 환경과 URL**
- 질문: "QA를 실행할 환경과 각각의 접속 URL을 알려주세요. 여러 환경이면 전부.
  (P0 발견 후보: <설정·배포 파일에서 발견한 URL 목록>)"
- 왜: 모든 케이스·시딩·헬스체크의 전제 — 환경을 잘못 잡으면 prod 오인 사고로 직결.
- 보조: ① P0 후보 제시 ② 모르겠다 → 정보성 — clearance 라우팅 ③ 전달 문구 생성.
- 기록: `environments.<env>.base_url`. 대상이 배포 환경으로 정해지면 로컬 기동은 생략하고
  base_url 헬스체크로 대체한다(P0의 [파악 못 한 것] 중 기동 실패 항목 해소).

**ENV-02 `[기본값형]` — 환경 등급**
- 질문: "각 환경의 등급을 확인해 주세요: <env>=<추정 등급>, … (prod / stage / dev)"
- 왜: prod는 유인+읽기전용 강제 — 등급이 모든 안전 계층의 기준값이다.
- 기본값: P0 추정값(URL·설정 근거). **모르겠다 → 보수 = prod로 간주**(읽기전용·무인 금지).
- 기록: `environments.<env>.grade`. 환경 판별 신호(P0 발견 fingerprint 후보)는
  `environments.<env>.fingerprint`에 확정 기록.

**ENV-03 `[기본값형]` — 무인 허용 환경**
- 질문: "무인 실행을 허용할 환경은 어디입니까? (스케줄 등록 자체는 나중에 [게이트4]에서 별도 허가)"
- 왜: 무인 안전 계층(allowlist·환경 fingerprint)의 대상 환경을 지정한다.
- 기본값: 비-prod `true` / **prod `false` 고정(변경 불가)**.
- 기록: `environments.<env>.unattended_allowed`

**ENV-04 `[기본값형]` — 데이터 경계**
- 질문: "각 환경에서 테스트 데이터를 만들어도 되는 범위는 어디까지입니까?
  (전용 네임스페이스·전용 계정·별도 스키마 등)"
- 왜: 실데이터 불가침(원칙 4)의 경계선 — 경계 밖 생성은 금지된다.
- 기본값: `test_prefix` 접두 리소스만 / prod는 생성 금지(읽기전용).
  모르겠다 → 보수 = 접두 리소스만 + P0에서 발견한 공용 공간은 생성 금지.
- 기록: `environments.<env>.data_boundary`

**ENV-05 `[기본값형]` — 타임존**
- 질문: "시간은 어떤 타임존으로 저장되고, 어떤 타임존으로 표시됩니까?
  (P0 발견: 저장=<…>, 표시=<…>)"
- 왜: 저장≠표시면 시간 경계 케이스(E)가 의무다 — 실전에서 반복 발생한 버그 유형.
- 기본값: P0 발견값. 발견 못 했으면 저장 UTC 가정으로 두고 케이스에서 실측 검증.
- 기록: `timezone.storage` / `timezone.display`

### 3.2 데이터

**DATA-01 `[기본값형]` — 테스트 접두**
- 질문: "생성하는 테스트 리소스의 식별 접두를 무엇으로 할까요?"
- 왜: 접두가 격리·정리·기계 차단의 공통 키다(원칙 4·5).
- 기본값: `qa-` (모든 생성 리소스에 run-id 병기).
- 기록: `data.test_prefix`

**DATA-02 `[기본값형]` — 실데이터 경계(불가침 대상)**
- 질문: "절대 건드리면 안 되는 실데이터(계정·리소스·패턴)가 있습니까? (P0 초안: <목록>)"
- 왜: 기계 차단 대상의 데이터 축 — 선언이 아니라 차단 목록으로 만든다.
- 기본값: 접두 없는 데이터 전체 불가침 + P0 초안 채택. 모르겠다 → 보수 = 초안 전량 채택.
- 기록: `data.deny_list`

**DATA-03 `[기본값형]` — 정리 방식**
- 질문: "테스트 데이터 삭제는 어떤 방식이 프로젝트 정책입니까? (soft-delete / hard-delete)"
- 왜: 정책과 다른 삭제는 그 자체가 사고다 — 임의 영구삭제 금지(원칙 5).
- 기본값: P0에서 발견한 프로젝트 삭제 정책.
  발견 못 했고 사용자도 모르면 → **정보성 — clearance 라우팅**(담당자 확인 항목,
  확인 전 정리가 필요하면 soft 우선).
- 기록: `data.cleanup_method`(정리 스크립트 절차는 phases/P2-artifacts.md, patterns/seeding.md).

**DATA-04 `[기본값형]` — 보존 정책**
- 질문: "soft-delete 잔재의 purge 주기와 `runs/` 리포트 보존 개수는 어떻게 할까요?"
- 왜: 정리 없는 반복 실행은 리소스·리포트를 무한 누적시킨다.
- 기본값: purge `30d` / runs 보존 `20`개(초과분 오래된 것부터 삭제).
- 기록: `data.retention.softdelete_purge` / `data.retention.runs_keep`

### 3.3 인증

**AUTH-01 `[필수입력형]` — 역할과 테스트 계정**
- 질문: "테스트할 역할(예: admin / member)과 역할별 테스트 계정이 있습니까?
  (P0 발견 역할 후보: <코드의 역할 enum·라우트 가드에서 추출한 목록>)
  계정 실값은 채팅에 적지 마시고, 안내드릴 `qa/.env.qa` 파일에 직접 넣어 주세요."
- 왜: 권한 케이스(D)의 축 — 역할이 없으면 권한 경계를 검증할 수 없다.
- 보조: ① P0 역할 후보 제시 ② 계정 없음/모르겠다 → 자격 — clearance 라우팅
  (계정 발급 요청 문구 생성) ③ 전달 문구 생성.
- 기록: `auth.roles[].name` / `auth.roles[].secret_ref` / `environments.<env>.account_secret_ref`
  — 키 이름은 에이전트가 제안(예: `QA_ADMIN_CRED`), 실값은 `.env.qa`에 사용자 직접 기입.

**AUTH-02 `[기본값형]` — 로그인 방식·SSO/2FA/캡차 대응**
- 질문: "로그인 과정에 SSO·2FA·캡차가 있습니까? (P0 발견: <…>)
  있다면 자동화 대신 '유인 로그인 → 세션 저장' 방식을 씁니다."
- 왜: 자동화 불가 영역을 지금 확정해야 P3 도중에 멈추지 않는다.
- 기본값: 일반 폼 로그인이면 `form`(자동) / SSO면 `sso` / 캡차·2FA 등이면 `manual`
  (둘 다 유인 로그인→세션 저장 절차는 동일 — `qa/.auth/<role>.state.json`, patterns/auth-roles.md).
  캡차 우회·자동 돌파 시도는 금지.
- 기록: `auth.roles[].login_method`

**AUTH-03 `[기본값형]` — 세션 수명과 갱신 수단**
- 질문: "로그인 세션은 얼마나 유지되고, 만료되면 어떻게 갱신합니까?"
- 왜: 프리플라이트 인증 체크와 무인 AUTH_EXPIRED 처리의 기준값이다.
- 기본값: `8h`(모르면 8h로 두고 실측 후 보정). `manual` 로그인인데 자동 갱신 수단이
  없으면 무인 범위 제약으로 config 주석에 기록.
- 기록: `auth.roles[].session_lifetime`

**AUTH-04 `[기본값형]` — 동시 세션 정책**
- 질문: "테스트 계정을 여러 사람·세션이 같이 씁니까? 동시 로그인하면 기존 세션이 끊깁니까?"
- 왜: 공유 계정의 세션 충돌은 플래키의 단골 원인 — 상호배제 정책을 정한다.
- 기본값: `single`(동시 세션 금지 — `qa/.lock` 상호배제와 함께 레이스 방지).
- 기록: `auth.concurrent_session_policy`

### 3.4 위험

**RISK-01 `[필수입력형]` — deny-list 확정**
- 질문: "다음은 코드에서 자동 추출한 불가역·위험 액션 후보입니다: <P0 초안 —
  부작용 있는 non-GET 라우트·UI 액션 목록, 항목별 권고 포함>.
  각 항목을 차단(무인 시 기계 차단)할지 허용할지 확정해 주세요."
- 왜: 오발사는 되돌릴 수 없다 — 이 답이 기계 차단 대상의 정본이 된다.
- 보조: ① P0 non-GET 자동 추출 초안 제시 ② 모르겠다 → 보수 = **초안 전량 차단**
  (과차단이 오발사보다 낫다 — 필요해지면 게이트에서 해제) ③ 판단이 안 서는 항목은
  담당자 전달 문구 생성.
- 기록: `risk.irreversible_actions`

**RISK-02 `[기본값형]` — 외부 부작용**
- 질문: "테스트 실행이 바깥에 흔적을 남기는 지점이 있습니까 —
  메일·알림·웹훅·과금 API·외부 시스템 리소스 생성? (P0 발견: <목록>)"
- 왜: 무인 반복은 부작용을 누적시킨다(원칙 7) — 차단할지 무해화할지 지금 정한다.
- 기본값: 발견 지점은 deny-list 편입 또는 무해화(테스트 수신함·샌드박스 키 등).
  모르겠다 → 보수 = 차단.
- 기록: `risk.irreversible_actions`(차단). 무해화에 자격이 필요하면(샌드박스 키 등)
  clearance.md 항목으로 추가.

**RISK-03 `[기본값형]` — run당 예산**
- 질문: "한 번의 run에서 외부 API 호출과 리소스 생성을 몇 건까지 허용할까요?
  (초과하면 중단하고 에스컬레이션합니다)"
- 왜: 폭주·무한루프의 기계적 상한선이다.
- 기본값: 외부 호출 `100` / 리소스 생성 `20`.
- 기록: `risk.budget_per_run.external_calls` / `risk.budget_per_run.resource_creations`

### 3.5 범위

**SCOPE-01 `[기본값형]` — 우선 기능**
- 질문: "우선 검증할 기능을 골라주세요. (P0 발견 기능 목록: <라우트·문서 기반 목록>)"
- 왜: 커버리지 규칙(우선 기능마다 A·B·D 최소 1케이스 — phases/P2-artifacts.md)의 입력.
- 기본값: P0 발견 핵심 기능 전부(케이스 총량 상한 내에서 실패 우선 배분).
- 기록: `scope.priority_features`

**SCOPE-02 `[기본값형]` — 제외**
- 질문: "이번 범위에서 뺄 기능·라우트가 있습니까? (제외 사유도 함께)"
- 왜: 명시적 제외가 없으면 나중에 커버리지 갭과 의도적 제외를 구분할 수 없다.
- 기본값: 없음. 단 deny-list 대상 실행 케이스는 자동 제외하고 "차단이 작동하는지"
  검증 케이스로 대체.
- 기록: `scope.excluded`(사유는 주석으로)

**SCOPE-03 `[기본값형]` — 케이스 총량 상한**
- 질문: "1차 케이스는 몇 개까지로 잡을까요?"
- 왜: 과밀 케이스는 유지비 폭탄 — 상한이 다이어트를 강제한다.
- 기본값: `40`.
- 기록: `scope.case_cap`

### 3.6 운영

**OPS-01 `[기본값형]` — 스케줄 의향**
- 질문: "안정화되면 무인 스케줄 운용까지 갈 생각이십니까?
  (지금은 의향만 — 실제 등록은 [게이트4]에서 별도 허가받습니다)"
- 왜: 의향에 따라 P4 코드화 우선순위와 러너 준비가 달라진다.
- 기본값: 미정(P4 안정화 후 결정). 어떤 답이든 게이트4 전 스케줄 등록 금지 —
  `ops.schedule.approved`는 `false` 유지.
- 기록: `ops.schedule`(cadence·window·env·trigger_method는 게이트4에서 확정)

**OPS-02 `[기본값형]` — 실패 정책**
- 질문: "무인 run이 연속으로 실패하면 몇 회째에 중단할까요?"
- 왜: 연속 실패 방치는 부작용만 누적시킨다 — 에스컬레이션 기준선.
- 기본값: 연속 2회 실패 시 중단+통보.
- 기록: `ops.failure_policy`

**OPS-03 `[기본값형]` — 통보**
- 질문: "run 결과 통보는 어떻게 받으시겠습니까?"
- 왜: 결과를 아무도 안 보면 무인 그린이 거짓 안심이 된다.
- 기본값: `file`(리포트 파일만) + `ack_hours: 24`(사용자가 24시간 내 확인 책임).
  푸시 채널은 프로젝트 선택 — 원하면 수단·자격을 clearance.md 항목으로 추가.
- 기록: `ops.notify.channel` / `ops.notify.ack_hours`

**OPS-04 `[기본값형]` — 리포트 형식**
- 질문: "리포트는 표준 형식(맨 위 3줄 요약 + 실패마다 해설)으로 드립니다.
  추가로 필요한 형식이 있습니까? (예: 경영 보고용 요약)"
- 왜: 읽히지 않는 리포트는 없는 것과 같다 — 수신자에 맞아야 한다.
- 기본값: 표준 형식만(templates/run-report.template.md).
- 기록: 표준이면 별도 기록 없음. 추가 요구는 `ops` 섹션 주석으로 남기고
  run-report 생성 시 반영.

## 4. 인터뷰 진행 절차

1. **선별**: discovery-report를 대조해 은행에서 질문을 고른다.
   발견된 항목은 확인 항목으로 전환(원칙 1). 조건부 질문(ENV-00)은 조건 충족 시에만.
2. **배치 구성**: 맨 앞에 `[필수입력형]`(별도 표기, 각각 후보 답 첨부) →
   이어서 카테고리 순서(환경 → 데이터 → 인증 → 위험 → 범위 → 운영)로 `[기본값형]`.
   "다 추천대로"로 기본값형 전부를 한 번에 수용할 수 있음을 안내한다.
3. **답 수집·라우팅**: "모르겠다"는 §2 규칙대로 보수 설정 또는 clearance 라우팅.
   보수 설정 적용 항목은 config 주석으로, clearance 라우팅 항목은 대장에 즉시 추가.
4. **qa-config.yaml 생성**: templates/qa-config.template.yaml 골격을 채워
   `qa/qa-config.yaml`로 생성. `meta.kit_version`(정본=킷 CHANGELOG.md)과
   `meta.config_schema_version` 스탬프. 시크릿 실값 기재 금지 — `secret_ref`만.
5. **.env.qa 안내**: 필요한 키 이름 목록(secret_ref와 일치)을 안내하고, 사용자가
   `qa/.env.qa`에 실값을 직접 기입하게 한다. 에이전트는 파일과 키의 **존재만** 확인
   (값을 읽거나 인용하지 마라). `.env.qa`와 `.auth/`가 qa/.gitignore(P0 초기화 산출물)에
   등록돼 있는지 확인·강제(없으면 qa/.gitignore를 생성/보완).
   - **형식 규격**: KEY=VALUE 한 줄 규격(.env 표준). 값에 공백·특수문자가 있으면 따옴표로 감싼다.
   - **자격증명은 분리 키가 표준**: 예 `QA_ADMIN_ID`, `QA_ADMIN_PW` — 한 키에 ID+PW를 합치지 않는다.
   - **Node 로드법**: `node --env-file=qa/.env.qa <스크립트>`.
6. **STATE.md 갱신**: 버전 스탬프·다음 액션·결정 로그(templates/STATE.template.md).

## 5. 사전 완결 (Upfront Clearance) — clearance.md 작성

원칙 12: 허가·자격·정보는 시작 전에 전부 확보한다. 테스트 도중 "담당자 회신 대기"로
멈추는 것은 실패다.

**작성 규칙** (templates/clearance.template.md 사용):
- P0의 준비물 후보 초안을 인터뷰 답과 대조해 확정하고 `qa/clearance.md`로 생성한다.
- 범위는 **P2~P5 전 과정** 필요분 전수: 테스트 계정·역할별 계정, API 키·시크릿,
  대상 환경 접근(VPN·방화벽·프록시), 외부 시스템 권한, 담당자 승인 필요 항목
  (데이터 정리 정책·스케줄 등).
- ID는 `CL-<번호>` 고정. 분류: 계정 / 키 / 접근 / 외부권한 / 승인.
- 항목마다 {무엇이·왜 필요한가, 요청 대상, **복붙 요청 문구**(부록에 전문), 상태, 기입값 위치}.
  기입값 위치: 시크릿이면 `.env.qa` 키 이름(config의 secret_ref와 일치), 아니면
  qa-config.yaml 필드명, 없으면 "(확인만 — 기입값 없음)".
- 상태 전이: 미요청 → 요청됨 → 확보 | 불가. 변경 즉시 대장 갱신 + STATE.md 캐시(확보 n/m) 동기화.
  **정본은 clearance.md** — STATE는 요약 캐시일 뿐이다.
- 인터뷰에서 자격·접근·정보성 "모르겠다"로 라우팅된 항목을 빠짐없이 편입한다.
  이미 확보된 것(예: `.env.qa` 기입 완료)은 즉시 "확보"로 표기.

**확보 대기와 P2 병행 규칙**:
- [게이트1] 승인 후에는 미확보 항목이 남아 있어도 **P2(산출물 생성)를 병행**한다 —
  담당자 회신을 기다리며 케이스·스크립트·런북을 만든다.
- 단, **접근 자격이 필요한 작업(UI 크롤링 등)은 해당 항목 확보 후로 연기**한다.
- **P3(라이브 실행)은 전 항목 확보 전 시작 금지** — 부분 확보로 시작하면 도중에 멈춘다.
  진입 시 STATE 캐시가 아니라 clearance.md(정본)를 재확인한다(phases/P3-first-run.md).
- "불가" 항목이 나오면: 의존 케이스를 수동 표기로 전환하고 축소 범위로 [게이트1] 재승인
  (templates/clearance.template.md의 P3 진입 조건).

## 6. [게이트1] 제시 규칙 — 설정 승인

templates/gate-summary.template.md 형식으로 제시하고 **반드시 멈춘다**(AGENT.md §4).

**구성**:
1. **지금 무엇을 승인하는가** (평문 1문장): "이 설정(계약)과 준비물 대장으로 QA를 준비합니다."
2. **승인하면 무슨 일이 일어나는가**: qa-config.yaml 확정(이후 변경도 게이트를 거침) ·
   P2 산출물 생성 시작 · 미확보 준비물의 요청 문구 전달(사용자 몫) 병행.
3. **위험 하이라이트** — 반드시 포함: prod 환경 등록 여부와 그 제약(유인+읽기전용) ·
   deny-list 요약(건수 + 대표 항목) · 무인 허용 환경 · 보수 설정이 적용된 항목 ·
   clearance 라우팅된 항목(P3 차단 예정임을 명시).
4. **에이전트 권고** → 응답 요청: **"승인" / "수정: <내용>" / "보류"**.

**YAML 평문 번역 규칙** (원문이 아니라 번역으로 제시한다 — 비전공자 기준):
- 카테고리 순서(환경 → 데이터 → 인증 → 위험 → 범위 → 운영)대로 각 1~3문장,
  "무슨 값 + 왜"로 풀어 쓴다. 전문용어는 풀이 병기
  (예: "deny-list = 실수로도 실행하지 않도록 기계적으로 차단하는 목록").
- 기본값 그대로인 항목은 묶어서 한 줄로, 사용자가 바꾼 값·보수 설정·라우팅 항목은 개별 명시.
- **원문은 qa/qa-config.yaml 경로로 첨부**한다(전문을 게이트 요약에 붙이지 않는다 — 요청 시 인쇄.
  번역이 정본을 대체하지 않는다).

**준비물 대장 포함**: 확보 현황(확보 n/m) 표 + 미요청 항목별 다음 행동
("부록 CL-XX 문구를 <요청 대상>에게 전달") 안내. 미확보가 있어도 게이트1은 통과 가능 —
단 P3 차단 예정을 함께 고지한다.

**응답 처리**:
- "승인" → STATE.md에 게이트1 통과(타임스탬프)·결정 로그 기록, 다음 액션 = "P2 시작
  (phases/P2-artifacts.md)". 미확보 항목 요청 발송을 사용자에게 재안내.
- "수정: <내용>" → config에 반영 후 바뀐 부분만 재제시(전체 재인터뷰 금지).
- "보류" → 진행 중단, STATE.md에 보류 사유 기록. 승인 없이 P2로 넘어가지 마라.
