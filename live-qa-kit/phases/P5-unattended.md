# P5-unattended.md — 무인 운용 (허가제·순환)

> **진입 조건**: P4 안정화 완료 — 러너 기준 **연속 2회 클린 런**(phases/P4-retro.md).
> **무인 범위 = `qa/runner/`에 편입된 케이스만.** 미편입 케이스는 유인 런에 남는다.
> 무인 스케줄은 **[게이트4] 명시 허가 없이는 절대 등록하지 않는다**(원칙 7·10).
> P5는 종착점이 아니라 **순환 상태**다: run 반복 + 정기 델타 디스커버리로 커버리지를 살아 있게 유지한다.

## 0. 시작 전 확인

- STATE.md 확인: 게이트3 통과 이력·러너 편입 케이스 목록·능력 프로파일(스케줄 수단·헤드리스 재호출 수단).
- 대상 환경: qa-config `environments`에서 `unattended_allowed: true`인 환경만 후보.
  `grade: prod`는 후보가 아니다 — prod는 유인+읽기전용만(설계 불변).
- 능력 프로파일에 스케줄 수단이 없으면 게이트4를 처음부터 "반무인" 조건으로 상정한다(§2 강등).

## 1. [게이트4] 스케줄 허가

`templates/gate-summary.template.md` 형식으로 제시하고 **멈춘다**. 승인 전 `ops.schedule.approved`는 false 고정 — 에이전트가 임의로 true로 바꾸지 않는다.

**허가 항목(전부 명시)**

| 항목 | 내용 | qa-config 기록처 |
|---|---|---|
| 주기 | 예: "매일 1회" | `ops.schedule.cadence` |
| 시간창 | 예: "02:00-04:00" | `ops.schedule.window` |
| 환경 | `unattended_allowed: true`인 환경 이름 | `ops.schedule.env` |
| 기동 방식 | cron / OS 작업 스케줄러 / 에이전트 스케줄 기능 — P0 능력 프로파일 기준 | `ops.schedule.trigger_method` |
| 중단 조건 | 연속 실패 N회(기본 2)·예산 초과·시간 상한·킬스위치(§5) | `ops.failure_policy` |

**3문항 고지 — 게이트 요약에 반드시 포함**

1. **무엇이 자동 실행되나**: 러너 편입 케이스 ID 목록·대상 환경·주기·생성되는 테스트 데이터(test_prefix 리소스).
2. **최악의 경우 무엇이 잘못될 수 있나**: 부작용을 솔직히 — 토큰 발급 누적·API 쿼터 소진·알림 발송·리소스 누적.
   allowlist·fingerprint가 **막는 것**과 **못 막는 것**(승인 라우트 안에서의 오동작 등)을 구분해 적는다.
3. **멈추는 법**: 원클릭 STOP 스크립트 위치·`qa/.stop` 직접 생성·스케줄 해제 방법 — 비전공자가 그대로 따라 할 수 있게.

**통보 확정**

- 기본 = `ops.notify.channel: file` — "통보는 `runs/<env>-<run-id>/report.md` 파일 생성뿐이며,
  사용자가 `ack_hours`(기본 24시간) 내 확인할 책임"임을 고지 문구에 포함해 동의받는다.
- 푸시 채널은 프로젝트 선택 — 채택 시 `ops.notify.channel` 갱신(게이트4에서 함께 승인).

**주기 > run 최대 소요시간 검증**

- 기준 실측 = P4 클린 런들의 최대 소요시간.
- 외부 하드 타임아웃 기본값 = 실측 최대 × 2(§5).
- **cadence 간격 > 하드 타임아웃**이어야 한다(겹침 방지 — `.lock`은 최후 방어일 뿐이다).
  불충족이면 주기를 늘리거나 케이스를 줄여 재상정한다.

**승인 후**: `ops.schedule` 갱신(approved: true 포함) → 스케줄 등록(§2) → STOP 스크립트 생성(§5) → STATE 결정 로그 append + 페이즈 P5 기록.
"수정: <내용>"이면 반영해 재상정, "보류"면 아무것도 등록하지 않고 P3↔P4 순환을 유지한다.

## 2. 실행 모델 — 2계층

**기본 계층 = 코드화된 run-all 러너(`qa/runner/`) — 에이전트 불요**

- 기동: OS 스케줄러(`trigger_method`)가 **스케줄 래퍼**를 호출. 래퍼의 책임 = 러너 기동 + 외부 하드 타임아웃(§5).
- 러너가 전부 **코드로** 수행: 프리플라이트(§3) → 시딩 → 케이스 실행·판정 → 증거 저장(redaction — patterns/evidence.md)
  → 정리+정리검증(patterns/seeding.md) → report 생성(§7) → 보존 정책 집행(§8).
- 러너는 시딩 시 STATE.md **잔여물 대장을 즉시 갱신**한다(생성 즉시 추가, 정리검증 후 완료 표기) —
  run이 도중에 죽어도 다음 세션이 대장만 보고 치울 수 있게(원칙 5·9).
- 러너는 Node.js 스크립트다 — 어떤 에이전트 생태계든 무인이 성립한다.

**선택 계층 = 헤드리스 에이전트 회고 — 재호출 수단이 있는 환경만**

- run 종료 후 래퍼가 에이전트를 헤드리스로 호출, 회고(phases/P4-retro.md 절차 축약)를 수행 — 실패 3분류·개선제안 작성.
- 호출 커맨드를 qa-config `ops.schedule.retro_command`에 기록한다(게이트4에서 함께 승인 — 이 필드는 선택 계층 채택 시에만 추가).
- 회고 계층도 **제안만** 한다 — 정책 변경은 물론 스크립트·셀렉터 수정도 무인 중에는 자동 적용 금지, 리포트에 제안으로 남긴다(§7).

**강등 — 반무인**

- 스케줄 수단·헤드리스 수단이 둘 다 없으면 "사람이 트리거하는 반무인": 사람이 러너를 수동 실행(더블클릭 또는 한 줄 커맨드), 이후 절차는 동일.
- 강등 사실을 게이트4 요약과 STATE에 명시하고 그렇게 보고한다(AGENT.md §5).

## 3. 프리플라이트 (러너 선두 — 하나라도 실패 시 케이스 0건 실행)

순서대로 체크한다:

1. **킬스위치**: `qa/.stop` 존재 → 시작하지 않고 STOPPED 기록. 사람이 파일을 지워야 재개된다.
2. **락**: `qa/.lock` 존재+신선(heartbeat 30분 이내) → **시작 거부** + 거부 사실 기록(다른 실행이 진행 중이다).
   stale(30분 초과) → 이전 실행의 잔여물 정리 후 **락 인수**(세션ID·모드·heartbeat 갱신).
3. **잔여물**: STATE 시딩 잔여물 대장에 미정리 행이 있으면 정리 + 잔여 0건 검증부터(원칙 5).
4. **헬스**: 대상 환경 `base_url` 헬스체크(qa/scripts/의 헬스체크 스크립트).
5. **도구**: Node.js·브라우저 바이너리 가용 확인.
6. **인증 유효**: 인증 필요 API 1콜. 만료 시 — 케이스를 실행하지 않고 run 전체를 **AUTH_EXPIRED로 스킵 + 통보**.
   FAIL로 세지 않는다(FAIL 오염 방지). 연속 실패 카운트에도 넣지 않는다.
   재개는 사람이 세션을 갱신해야 한다(patterns/auth-roles.md) — 그때까지 매 run 통보가 반복된다.
7. **환경 fingerprint**: §4 검증 — 불일치 시 즉시 중단+에스컬레이션.

## 4. 안전 강제 계층 (선언이 아니라 기계로)

- **allowlist 기본 자세**: 무인 run은 **승인 라우트 외 모든 non-GET을 차단**한다.
  승인 라우트 = 게이트2·3을 통과한 케이스가 명시적으로 사용하는 라우트 목록 — 러너에 코드로 내장한다.
  - Playwright 경로: `context.route()`로 **네트워크 레벨** 차단.
  - 대화형 경로(회고 계층 등에서 브라우저를 쓰는 경우): **클릭 전 파괴적 UI 휴리스틱 검사**
    (삭제/발송/결제류 라벨·확인 다이얼로그·`risk.irreversible_actions` 패턴 — patterns/browser.md).
  - 미등록 non-GET 발견 시: 해당 케이스 **스킵 + 에스컬레이션**. 차단 발동은 전부 리포트에 기록한다(§7).
    차단이 잦아지면 표면이 변한 신호다 — 델타 디스커버리를 앞당겨 상정하라(§6).
- **deny-list**: `risk.irreversible_actions`는 유·무인 공통 차단. allowlist는 그 위에 얹는 무인 전용 기본 자세다(원칙 4).
- **환경 fingerprint 검증**: 대상 URL과 응답 서명(헬스 응답의 환경 식별값·버전 정보 등)이
  qa-config `environments`의 해당 env 정의와 일치하는지 run 시작 시 확인한다.
  불일치 = prod 오인 가능성 — **즉시 중단 + 에스컬레이션**, 케이스 실행 금지.

## 5. 한도·중단 (폭주 방지)

| 장치 | 명세 |
|---|---|
| run당 예산 | `risk.budget_per_run` — external_calls·resource_creations를 러너가 카운트. 초과 즉시 중단 → 정리 → 에스컬레이션 |
| 시간 상한 | 러너 내부 총 실행시간 상한 + **스케줄 래퍼의 외부 하드 타임아웃(프로세스 트리 kill)** — 러너 자체가 행에 걸려도 죽는다. 하드 타임아웃 기본 = P4 실측 최대 소요 × 2(§1 검증과 연동) |
| 킬스위치 | `qa/.stop` 플래그 파일. 체크 시점: **케이스 경계마다 + 스크립트 내부 루프 폴링**(readiness 폴링·재시도 루프가 .stop을 놓치지 않게). 발견 시: 진행 중 케이스 안전 종료 → 잔여물 정리 → STOPPED 기록 + 통보 |
| 원클릭 STOP | 더블클릭용 스크립트를 qa/scripts/에 생성 — Windows용 `.cmd`와 POSIX용 `.sh` 둘 다(bash 보편성 가정 배제). 내용 = `qa/.stop` 생성 한 줄. 위치·사용법을 게이트4 고지 3번에 명시 |
| 연속 실패 에스컬레이션 | `ops.failure_policy`(기본: 연속 2회). 러너가 직전 run 결과를 읽어 카운트 — 도달 시 **스스로 `qa/.stop`을 생성**해 이후 run을 막고 통보. 재개 = 사람이 원인 확인 후 `.stop` 제거 |

STOPPED·에스컬레이션·재개는 발생 즉시 STATE 결정 로그에 남긴다(러너 발생분은 리포트에 기록하고, 다음 에이전트 세션이 STATE로 옮긴다).

## 6. 순환 — 델타 디스커버리 (커버리지 노후화 방지)

무인 그린이 "표면은 바뀌었는데 옛 케이스만 도는" 거짓 안심이 되지 않게 한다.

- **주기**: 게이트4에서 스케줄과 함께 확정. 권고 기본 = 배포 감지 시(감지 수단이 없으면 주 1회).
- **절차**(에이전트 세션에서 수행 — 유인 또는 헤드리스 회고 계층):
  1. **P0 축약판**: phases/P0-discovery.md 중 라우트/API 표면 파악만 재실행(질문 금지 동일).
  2. STATE의 "마지막 디스커버리 기준"(커밋 SHA/앱 버전)과 표면 **diff** — 신규/변경/삭제 라우트.
  3. diff를 케이스 인벤토리와 대조해 **커버리지 갭 리포트** 작성 — 커버되지 않은 신규 표면, 사라진 표면을 여전히 치는 케이스.
  4. **게이트 상정** — 케이스 추가/수정/폐기는 사람이 결정한다(원칙 10). 승인된 신규 케이스는
     P2(작성)→P3(유인)→P4(코드화) 경로를 거쳐야 러너에 편입된다 — **무인 직행 금지**.
  5. STATE의 마지막 디스커버리 기준 갱신(SHA/버전·실행일).
- **커버리지 신선도**: 매 run-report에 표기 — 마지막 디스커버리 기준과 경과 일수. 오래되면 리포트가 스스로 경고하게 한다.

## 7. 매 run 산출

`runs/<env>-<run-id>/`에:

- **report.md**(templates/run-report.template.md): 맨 위 3줄 요약(신호등·조치 필요 여부·비전공자용 한 문장) +
  정량 필드(케이스 수/PASS/FAIL/SKIP/플래키·실패 3분류 집계·실행 시간) + **차단 발동 기록**(§4) + **커버리지 신선도**(§6).
- **evidence/<case-id>/**: 증거 — redaction 완료본만 저장(원칙 6, patterns/evidence.md).
- **전회 diff**: 직전 run 대비 신규 실패·해소·플래키 전환.
- **개선제안**(회고 계층이 있을 때): 기술 세부(셀렉터·대기 보정)도 무인 중에는 제안+근거로만.
  **정책 변경 — 케이스 추가/제외·기준 변경·known-issue 등록 — 은 자동 적용 금지**(원칙 10),
  제안으로 적어 두고 다음 게이트(사람 세션)에서 승인받는다. 러너 단독 계층이면 관찰 기록만 남긴다.

## 8. 리소스 보존 정책 집행 (누적 방지)

러너 정리 단계에 코드로 편입한다. P1에서 확정한 `data.retention` 값을 그대로 집행하며 러너가 임의로 바꾸지 않는다:

- **runs/ 보존**: `data.retention.runs_keep`(기본 20) 초과분을 오래된 run 디렉터리부터 삭제.
- **soft-delete 잔재 purge**: test_prefix 리소스 중 soft-delete 상태로 남은 것을
  `data.retention.softdelete_purge`(기본 30d) 도달 시 purge — 프로젝트 삭제 정책이 우선(원칙 5). 실행·건수는 리포트에 기록.
- 정리 방식은 qa-config `data.cleanup_method`를 참조한다(soft | hard — P1 확정 정책값).
- 집행 대상은 **test_prefix + run-id로 식별되는 리소스와 qa/ 산출물만**(원칙 4) — 접두 없는 데이터는 건드리지 않는다.

## 9. 체크리스트

**설치(게이트4 승인 직후)**

- [ ] `ops.schedule` 갱신(approved: true·cadence·window·env·trigger_method) — 선택 계층 채택 시 `retro_command` 포함
- [ ] 스케줄 래퍼 생성(기동 + 외부 하드 타임아웃) 후 `trigger_method`로 등록
- [ ] 원클릭 STOP 스크립트 생성(qa/scripts/) + 동작 확인(`.stop`이 실제 생성되는지)
- [ ] 킬스위치 리허설: `.stop` 생성 → 러너 기동 → 시작 거부 확인 → `.stop` 제거
