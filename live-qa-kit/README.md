# README — Live QA Kit (사람용 안내서)

> 이 문서는 사람이 읽는 1페이지입니다. IT 비전공자 기준으로 썼습니다.
> AI 에이전트에게 줄 문서는 `AGENT.md`이고, 여러분이 그걸 읽을 필요는 없습니다.

## 이게 뭐 하는 물건인가요?

Live QA Kit은 AI 에이전트에게 "우리 웹 서비스가 실제로 잘 동작하는지" 검사를 맡기는 팀 공용 절차 모음입니다. 이 폴더를 프로젝트에 복사하고 한 줄만 입력하면, 에이전트가 프로젝트를 스스로 분석하고 → 꼭 필요한 것만 질문하고 → 테스트를 만들어 실행하고 → 사람이 읽을 수 있는 리포트를 씁니다. 여러분은 몇 가지 질문에 답하고 중요한 길목(게이트)에서 승인만 하면 되며, 여러분의 허가 없이는 어떤 것도 자동으로 반복 실행되지 않습니다.

## 시작 전 체크리스트

| # | 확인할 것 | 안 되면 — 개발팀에 이 문장을 보내세요 |
|---|---|---|
| 1 | **AI 에이전트가 실행되나요?** 에이전트 채팅창을 열어 "안녕"이라고 입력했을 때 답이 옵니다. | "AI 코딩 에이전트(Claude Code 등)를 제 PC에서 쓸 수 있게 설치·설정해 주세요. 채팅창에서 파일을 읽고 쓸 수 있어야 합니다." |
| 2 | **검사할 프로젝트의 코드 폴더가 어디 있는지 아나요?** 그 폴더를 에이전트로 열 수 있어야 합니다. | "QA 대상 프로젝트의 코드 폴더 경로를 알려 주시고, 제 PC에서 열 수 있게 해 주세요." |
| 3 | **이 킷을 붙여넣었나요?** `live-qa-kit/` 폴더 전체를 프로젝트 코드 폴더 바로 아래에 복사합니다. | "live-qa-kit 폴더를 우리 프로젝트 최상위 폴더 바로 아래에 복사해 주세요." |
| 4 | **검사할 환경의 주소(URL)를 아나요?** 몰라도 시작은 되지만, 에이전트가 곧 물어봅니다. | "라이브 QA를 실행할 환경(스테이징 등)의 접속 주소와, 테스트용 계정이 있는지 알려 주세요." |

## 시작법 — 한 줄이면 됩니다

에이전트 채팅창에 이렇게 입력하세요.

```
live-qa-kit/AGENT.md 읽고 시작해
```

나머지는 에이전트가 안내합니다. 중간에 대화가 끊겨도 같은 한 줄을 다시 입력하면 하던 곳부터 이어집니다.

## 이후 사람이 하는 일 5가지

1. **질문에 답하기** — 판단이 필요한 질문에는 전부 추천 기본값이 있어서 "다 추천대로 해"라고 해도 됩니다. 직접 입력해야 하는 사실 질문(환경 주소, 테스트 계정 등)은 2~3개뿐입니다.
2. **준비물 확보하기** — 에이전트가 준비물 대장(`qa/clearance.md`)에 항목별로 "복붙 요청 문구"를 만들어 줍니다. 그대로 담당자에게 전달하고, 회신을 받으면 에이전트에게 알려 주세요. 비밀번호·키 같은 값은 채팅에 적지 말고 `qa/.env.qa` 파일에 직접 기입합니다(방법은 에이전트가 안내).
3. **게이트 승인하기** — 중요한 길목마다 에이전트가 "무엇을 승인하는가 → 승인하면 무슨 일이 일어나는가 → 위험 → 권고" 형식으로 요약해 멈춥니다. **"승인" / "수정: <내용>" / "보류"** 셋 중 하나로만 답하면 됩니다.
4. **리포트 읽기** — 실행이 끝나면 `qa/runs/<환경>-<실행번호>/report.md`가 생깁니다. 맨 위 3줄 요약(전체 신호등 · 조치 필요 여부 · 한 문장 설명)만 봐도 충분하고, 실패 항목마다 "무슨 뜻인가 · 심각도 · 권장 다음 행동"이 붙어 있습니다.
5. **멈추는 법 알아두기(STOP)** — 무인 운용을 허가하면 에이전트가 `qa/` 폴더에 더블클릭용 STOP 스크립트를 만들어 줍니다. 그걸 더블클릭하면(또는 `qa/` 폴더에 `.stop`이라는 빈 파일을 만들면) 진행 중인 자동 실행이 안전한 지점에서 멈춥니다.

## FAQ — 이럴 땐 이렇게

**Q. 시딩이 뭐예요?**
테스트에 필요한 데이터(예: 테스트용 게시글·계정)를 실행 전에 미리 만들어 두는 일입니다. 전부 `qa-` 접두가 붙은 식별 가능한 데이터만 만들고, 끝나면 지우고 "지웠음"까지 검증합니다. 실제 운영 데이터는 건드리지 않습니다.

**Q. 게이트가 뭐예요?**
사람이 승인해야 다음으로 넘어가는 지점입니다. 총 4곳 — ① 설정 승인 ② 테스트 케이스 승인 ③ 첫 실행 결과 승인 ④ 무인 스케줄 허가. 에이전트는 게이트에서 반드시 멈추고, 여러분 승인 없이는 넘어가지 않습니다.

**Q. 무인 운용은 안전한가요?**
무인은 허가제입니다(게이트4에서 주기·환경·멈추는 법까지 고지받고 허가). 허가 후에도 승인된 동작 외의 쓰기·삭제는 기계적으로 차단되고, 실행당 예산·시간 상한이 있으며, 운영(prod) 환경에서는 무인 실행 자체가 금지입니다. 그리고 언제든 STOP으로 멈출 수 있습니다.

**Q. 멈추고 싶어요.**
대화 중이면 채팅에 "보류" 또는 "멈춰"라고 입력하세요. 무인 실행 중이면 `qa/` 폴더의 STOP 스크립트를 더블클릭하세요(위 5번). 멈춘 이유와 상태는 파일로 남아서, 나중에 이어서 할 수 있습니다.

**Q. 실패가 났어요. 우리 서비스에 버그가 있는 건가요?**
아직 모릅니다 — 실패는 ① 제품 버그 ② 테스트 자체의 버그 ③ 문서·설정 문제 셋 중 하나이고, 에이전트가 근거를 확인해 분류합니다(단정 전에 재현 2회, 설계 의도 확인). 리포트의 해당 실패 항목에서 "무슨 뜻인가 · 심각도 · 권장 다음 행동"을 읽고, 제품 버그로 확정되면 게이트에서 등록 승인을 요청받게 됩니다.

## 용어집

| 용어 | 한 줄 설명 |
|---|---|
| 시딩(seeding) | 테스트에 필요한 데이터를 실행 전에 미리 만들어 두는 일. 끝나면 지우고 지웠음을 검증 |
| 게이트(gate) | 사람이 "승인/수정/보류"로 답해야 다음으로 넘어가는 승인 지점(총 4곳) |
| 케이스(case) | 테스트 1건. `A-01`처럼 분류-번호 ID가 붙는다(A=실패·경계, B=핵심 동작, D=권한 등) |
| evidence(증거) | PASS/FAIL 판정의 근거 파일(요청·응답·화면·DB 확인). `runs/.../evidence/<케이스ID>/`에 저장 |
| run | 테스트 1회 실행. 결과는 `qa/runs/<환경>-<실행번호>/`에 리포트+증거로 남는다 |
| clearance(준비물 대장) | 시작 전에 확보할 계정·키·권한·승인 목록. 전 항목 확보 전에는 라이브 실행을 시작하지 않는다 |
| 러너(runner) | 에이전트 없이도 도는 자동 실행 프로그램. 무인 운용은 러너에 편입된 안정 케이스만 수행 |
| 킬스위치(.stop) | `qa/.stop` 파일이 생기면 자동 실행이 안전한 지점에서 멈추는 장치. STOP 스크립트가 이 파일을 만든다 |
