# auth-roles.md — 로그인 전략 · 세션 규약 · 역할 전환 · 권한 경계

> 인증이 걸린 작업(로그인, 세션 확보·갱신, 권한 케이스 D)을 할 때 로드한다.
> 역할·전략·secret_ref의 정본은 qa-config.yaml `auth:` 절(templates/qa-config.template.yaml),
> 시크릿 취급은 원칙 6(principles.md). 브라우저 조작 바인딩은 patterns/browser.md.
> 사용처: 프리플라이트 인증 체크(phases/P3-first-run.md §1) · AUTH 이슈 분류(phases/P4-retro.md §1) · 무인 프리플라이트(phases/P5-unattended.md).

## 1. 로그인 전략 4유형 — 역할마다 하나를 확정

P0에서 인증 방식을 발견해 후보를 제시하고, P1에서 역할마다 확정해
qa-config `auth.roles[].login_method`(form|sso|token|manual)에 기록한다.

| 전략 | login_method | 방법 | 언제 |
|---|---|---|---|
| 테스트 계정 직로그인 | `form` | ID/PW 폼을 스크립트로 채워 로그인. 자격은 `secret_ref`가 가리키는 `.env.qa` 키에서 읽는다(실값을 파일·로그에 남기지 마라) | 로그인 폼이 있고 캡차·2FA가 없을 때 — 가장 안정, 1순위 |
| dev 우회 | `token` | 프로젝트가 공식 제공하는 테스트용 토큰·세션 발급 수단(dev 전용 엔드포인트, 헤더 주입 등)으로 로그인 UI 없이 인증 | 대상이 dev/stage이고 우회 수단이 공식일 때. prod 금지 |
| SSO 유인 로그인 → 세션 재사용 | `sso` | 실 SSO는 자동화하지 않는다. 유인 headed 로그인 1회(§3) → storageState 저장 → 세션 수명 동안 재사용 | 조직 IdP 경유 로그인만 가능한 환경 |
| 유인 로그인 → 세션 재사용 | `manual` | 절차는 sso와 동일 — 자동 로그인을 막는 사유가 SSO가 아닌 다른 것(캡차·2FA·메일 링크 등)일 때 | 자동 로그인은 불가하나 세션 재사용은 가능할 때 |

- **자동화 불가 → 수동 표기**: 유인 로그인→재사용조차 성립하지 않으면(세션 즉시 만료·디바이스 바인딩·매 요청 재인증 등) 그 역할에 의존하는 케이스를 **수동 표기**(자동·무인 범위 제외)로 전환하고, 축소 범위로 **[게이트1] 재승인**을 받아라 — clearance.md의 "불가" 항목 처리와 같은 규칙이다.
- 캡차·2FA는 **기계로 우회하지 않는다.** 사람 개입(§7)으로 통과시키거나 수동 표기한다.

## 2. 스크립트 모드 세션 규약 — `qa/.auth/<role>.state.json`

원칙 6("시크릿은 어디에도 남기지 않는다")의 **유일한 예외**다. 아래 규약을 전부 지킬 때만 허용된다.

- **형식 = Playwright storageState**(`{cookies, origins}`). Playwright 경로와 runner(P5)의 공용 정본이며, `<role>`은 qa-config `auth.roles[].name`과 일치시킨다.
- **gitignore 강제**: 파일을 만들기 **전에** qa/.gitignore에 `.auth/`가 등록돼 있는지 확인하라(P0 초기화 산출물 — 없으면 qa/.gitignore를 생성/보완). 커밋된 세션 파일은 유출이다.
- **만료시각 기록**: 로그인 성공 시각 + qa-config `session_lifetime`으로 계산해 JSON 최상위 `qa_expires`(ISO 8601) 키로 기입한다. Playwright는 cookies/origins만 읽으므로 로드에 지장 없다. 단 `qa_expires`는 힌트다 — 유효 판정의 정본은 §4의 유효성 체크 1콜.
- **인용 금지**: 파일 내용(쿠키·토큰 값)을 로그·리포트·evidence·채팅 출력·커밋 어디에도 인용하지 마라. 경로 언급은 가능, 내용 인용은 금지.
- 대화형 브라우저 도구 경로는 도구 브라우저의 로그인 세션을 그대로 쓰므로 이 파일이 필수는 아니다 — 단 runner(P5)·Playwright 스크립트·API 직콜(§6)이 쓰려면 필요하다.

## 3. 유인 headed 로그인 → 내보내기 절차 (sso·manual 공통, form 실패 시 폴백)

1. **headed 브라우저 기동**: Playwright는 `headless: false`로(설치·headed 규칙은 patterns/browser.md), 대화형 도구는 로그인 페이지로 이동만.
2. **개입 요청**: §7 고정 형식으로 사람에게 요청한다. 요청 시점은 P3 사전 고지(phases/P3-first-run.md §2)에 미리 포함돼 있어야 한다 — 갑자기 사람을 찾지 마라.
3. **완료 확인**: 사람이 "완료"라 답하면 유효성 체크 1콜(§4)로 로그인 성공을 확인한다. "화면이 바뀌었다"로 판정하지 마라(원칙 2).
4. **내보내기**: Playwright — `context.storageState({ path: 'qa/.auth/<role>.state.json' })`. 대화형 도구 경로에서 파일이 필요하면(runner 준비 등) 이 절차를 Playwright headed로 다시 수행해 내보낸다 — 도구 세션에서 쿠키를 긁어내려 하지 마라(httpOnly 접근 불가, 유출 위험).
5. **마무리**: `qa_expires` 기입 + gitignore 확인(§2). 세션 확보 사실만 기록하고 내용은 어디에도 남기지 않는다.

## 4. 세션 유효성 체크 · 만료 갱신

**유효성 체크 — 프리플라이트 표준**(모든 run 공통: phases/P3-first-run.md §1 #3, phases/P5-unattended.md 프리플라이트):

- **인증이 필요한 가벼운 API 1콜** — GET·부작용 없음·빠른 응답(현재 사용자 조회류가 이상적). 후보는 P0 라우트 표면에서 정하고 P2에서 `scripts/`의 체크 스크립트로 만든다.
- 판정: **2xx = 유효** / **401·403 = 만료·무효** / **5xx·타임아웃 = 환경 문제** — 마지막 것을 AUTH로 오판하지 마라(헬스 문제는 프리플라이트 #1의 몫).
- qa-config `auth.roles`의 **역할별로 각각 1콜**.

**만료 갱신**:

- 유인 run: `form`|`token`은 자동 재로그인 → state 재저장. `sso`|`manual`은 §3 유인 절차 재수행. 만료로 실패한 케이스는 FAIL로 세지 말고 **AUTH 이슈**로 분류한 뒤 세션 재확보 후 재실행하라(phases/P4-retro.md §1).
- 무인 run(P5): **갱신을 시도하지 않는다.** 프리플라이트에서 만료를 감지하면 케이스 실행 없이 **AUTH_EXPIRED로 스킵 + 통보**(FAIL 오염 방지). 세션 재확보는 다음 유인 세션의 몫이다.
- 공유 계정: qa-config `concurrent_session_policy: single`이면 같은 계정의 세션을 동시에 쓰지 마라 — 역할 전환은 순차(§5), 사람이 그 계정으로 로그인 중인지도 run 시작 전에 확인하라(한쪽 로그인이 다른 쪽 세션을 끊는 앱이 있다).

## 5. 역할 전환 테스트 패턴 — D 케이스의 축

- **역할별 계정이 D 케이스의 전제다**: 최소 2역할(높은 권한 + 낮은 권한). 1역할뿐이면 D 커버리지 한계를 케이스 인벤토리에 명시하고 clearance.md로 추가 계정을 요청하라(원칙 12).
- **전환 방법**: Playwright — 역할별 **별도 browser context**에 각자의 `qa/.auth/<role>.state.json`을 로드. 대화형 — 로그아웃 → 재로그인으로 순차 전환하고 전환 시점을 케이스 기록에 남긴다. `single` 정책이면 동시 컨텍스트 금지 — 순차로.
- **실행 대상은 시딩 리소스만**: 권한 시도가 뚫리면 그 액션은 실제로 실행된다. 타인 리소스 접근·삭제 시도는 반드시 `qa-` 접두 시딩 리소스를 대상으로 하라(patterns/seeding.md) — 실데이터에 권한 시도 금지(원칙 4).

표준 3패턴 — 우선 기능마다 최소 1건(phases/P2-artifacts.md 커버리지 규칙):

| 패턴 | 세팅 → 실행 | 기대 |
|---|---|---|
| 타인 리소스 접근 | 역할 A(소유자)로 리소스 시딩 → 역할 B로 조회·수정·삭제 시도 | 거부. 소유자 역할과 검증 역할을 케이스에 명기 |
| 관리자 전용 | 일반 역할 세션으로 관리자 전용 기능·라우트 호출 | 거부 — UI와 API 양 레벨(§6) |
| 존재 마스킹 (403 vs 404) | 낮은 권한으로 접근 불가 리소스 조회 → 상태코드 확인 | **기대값은 설계 문서가 정본**(원칙 8 — 403 자체가 버그는 아니다). 설계가 존재 은닉이면 403은 "존재를 노출하는" 정보 노출 버그 후보, 404가 정답 |

- 판정은 실제값으로: 상태코드 + 응답 body(에러 메시지에 타인 데이터·내부 정보가 새는지도 증거 대상).

## 6. UI 권한 ≠ API 권한 — 분리 검증 의무

**화면에서 안 보인다 ≠ 막혀 있다.** 버튼 숨김은 프론트 처리일 뿐, 백엔드 경계의 증거가 아니다.

- D 케이스는 두 레벨을 **각각** 검증한다: ① UI 레벨 — 해당 역할 화면에서 미노출·비활성 확인 ② API 레벨 — 같은 역할의 세션으로 그 API를 **직접 호출**해 4xx 거부 확인. **UI 확인만으로 D 케이스를 PASS 처리하지 마라.**
- 최소 규칙: UI에서 숨겨진 액션마다 대응 API 직콜 검증 1건.
- 직콜 방법: Playwright — `request.newContext({ storageState: 'qa/.auth/<role>.state.json' })`로 호출. 대화형 — 페이지 컨텍스트 JS fetch 평가 또는 Node 스크립트(`scripts/`)가 state.json을 읽어 인증을 구성(값 인용 금지 규약은 그대로).
- 단 deny-list 규칙은 권한 검증에도 그대로 적용된다 — 불가역 액션으로 등록된 라우트는 "거부될 것"이라는 기대만으로 호출하지 마라(qa-config `risk.irreversible_actions`, 원칙 4).
- 증거: 요청 URL·상태코드·응답 body를 `evidence/<case-id>/`에(스키마·redaction은 patterns/evidence.md — Authorization/Cookie 헤더 제거는 기본).

## 7. 개입 요청 형식 — 수동 로그인·2FA

사람에게 개입을 요청할 때는 **왜 필요한지 + 정확히 뭘 하면 되는지 + 완료 신호**를 고정 형식으로 제시하라. 개입 시점은 P3 사전 고지 목록에 미리 포함한다.

```
[개입 요청 — <role> 로그인]
왜: <근거 — 예: 프리플라이트 유효성 체크 401, 세션 만료>.
    이 세션이 없으면 <의존 케이스 ID — 예: D-01, D-03>을 실행할 수 없습니다.
무엇을:
  1. 지금 띄운 브라우저 창에서 <role> 계정으로 로그인해 주세요.
     (계정 정보는 qa/.env.qa 의 <secret_ref> — 직접 확인해 입력)
  2. 2FA·캡차가 나오면 화면에 직접 입력해 주세요.
     비밀번호·인증 코드를 채팅에 적지 마세요 — 저는 받지 않습니다.
  3. 로그인이 끝나 <완료 기준 — 예: 대시보드 화면>이 보이면 "완료"라고 답해 주세요.
다음: 제가 세션 유효성을 확인(§4)하고 저장(§3)한 뒤 <재개 지점>부터 계속합니다.
예상 소요: 약 <n>분
```

- **비밀번호·2FA 코드를 채팅으로 받지 마라**(원칙 6). 입력은 사람이 브라우저에 직접 한다.
- 개입 완료 후: 유효성 체크 1콜 → state 저장·`qa_expires` 기입 → 중단 지점부터 재개. 이 개입은 케이스 실패가 아니다 — AUTH 이슈로만 집계한다(§4).
